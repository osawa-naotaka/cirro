# プロジェクト憲章 - Cirro

## 1. プロジェクト基本情報

### 1.1 プロジェクト名
**Cirro** - React の島（islands）アーキテクチャによる、CSP 厳格（`unsafe-inline` 不要）を実現する軽量 static site generator (SSG)

### 1.2 プロジェクト実行の背景・課題

#### 最大の動機：CSP `script-src 'self'` を達成できる SSG が欲しい
本プロジェクトの出発点は、既存ツール（特に Astro）への具体的な不満である。

- Astro はスクリプトの大半を外部ファイルに分離できるが、**ごくわずかにインラインスクリプトが HTML に残る**。そのため Content Security Policy で **`unsafe-inline` を許可せざるを得ず**、`script-src 'self'` だけの厳格な CSP を達成できない。
- インラインスクリプトを **完全にゼロ** にし、`unsafe-inline` なしの厳格な CSP を当然のように満たせる SSG を作りたい。これが Cirro の存在意義であり、最大の差別化点である。

#### その他の課題
- Next.js などの SSG は規模が大きく、簡単なサイトを構築するには複雑すぎる
- Astro は多フレームワーク対応で汎用的だが、独自の `.astro` 構文の学習が必要。**React だけに絞り**、`.astro` を学ばずに React の知識だけでサイトを構築したい

### 1.3 プロジェクトの目的

#### プロジェクトの目的
- インラインスクリプトを一切生成せず、**`script-src 'self'`（`unsafe-inline` 不要）の厳格な CSP** を満たす静的サイトを生成する（第一の目的）
- 軽量な SSG を blog やポートフォリオサイトを運営したい人に提供する
- 静的サイトを Cloudflare にデプロイすることを第一目標とする
- React だけで、クライアント側でリアクティブな要素（島）を記述できるようにする

#### 「軽量」の定義
Cirro が目指す「軽量」は、バイトと開発体験の両面を指す。

- **バイト面**: 本文などの静的部分はビルド時に HTML 化し、クライアントへ JS を送らない。インタラクティブな「島」を含むページにのみ、React ランタイムと島のコードが乗る。島が1つもないページは **JS ゼロ** にできる
- **DX 面**: Next.js に比べて設定・概念がシンプルで認知負荷が低く、ビルドが速く、規約が少なく把握しやすい。React 以外の独自構文を学ぶ必要がない

#### 概要
- Vite ベースで実装する。Cirro 自体は単一の Vite プラグインではなく、独自のビルド処理・島プラグイン・CLI・規約を束ねた薄いフレームワークとして提供する
- TypeScript を利用して型を活用する（特にルーティング設定の型付けを売りとする）
- React の島（islands）アーキテクチャで SSG を実現する。React Server Components / Flight には依存しない
- インタラクティブな箇所は React の「島」として記述し、その箇所だけクライアントでハイドレートする

#### 技術的な土台
React 標準の `react-dom/server` と `react-dom/client` のみで、RSC を使わずに島アーキテクチャを実現する。

- **ビルド時（サーバーレンダリング）**
  - 各ページを `react-dom/server` の `renderToStaticMarkup` 等で **純粋な静的 HTML** に変換する（ハイドレーション用マーカーを持たない）
  - データ取得は、HTML 化の前にビルドスクリプト側で行う
- **島（islands）の宣言と出力**
  - インタラクティブなコンポーネントは `<Island>`（`client:load` 相当の宣言 API）で囲む
  - ビルド時に、島は「プレースホルダ要素 + `data-*` 属性（props を JSON で格納）+ 外部エントリ JS への `<script src>`」へ展開される
  - **props はインラインスクリプトではなく `data-*` 属性で渡す**ことで、インラインスクリプトの発生を回避する
- **クライアント（ハイドレーション）**
  - 島ごとに独立した `hydrateRoot`（初期 HTML を活かす場合）または `createRoot`（プレースホルダから構築する場合）でマウントする
  - 本文（静的部分）はハイドレートしないため、本文のコードもランタイムもクライアントに送られない
- **CSP 厳格性の担保**
  - 上記により、生成物には**インラインスクリプトが一切含まれない**。すべての JS は外部ファイル（`<script src>`）として読まれ、`script-src 'self'` のみで成立する

## 2. プロジェクトスコープ

スコープは段階的に拡張する。Phase 1 を確実に完成させることを最優先とする。

### 2.1 Phase 1（初期バージョン・第一目標）— 島 SSG + 厳格 CSP

#### 機能要件
- **インラインスクリプトゼロ / 厳格 CSP**（最重要・差別化の核）
  - 生成される全ページがインラインスクリプトを含まず、`script-src 'self'` の CSP で動作する
- **Config Base Routing**
  - `.ts` ファイルに JavaScript オブジェクトとしてルーティング情報を直接書き込む
  - ルート定義オブジェクトに型を付け、型安全なルーティングを提供する
  - 動的ルート（例: `/posts/[slug]`）は、ルート定義からパラメータ一覧を返す仕組みで表現する
- **島（islands）アーキテクチャ**
  - 本文は `renderToStaticMarkup` で静的 HTML 化
  - インタラクティブ要素は `<Island>` で宣言し、その箇所だけ `hydrateRoot` / `createRoot` でハイドレート
  - props は `data-*` 属性で受け渡す
- **コンテンツ層**
  - プレーンな Markdown を記事コンテンツとして扱えるようにする（MDX は採用しない。理由は 2.3 を参照）
- **開発体験**
  - dev サーバーと HMR
- **配布形態**
  - MPA（クライアント遷移を伴わない、各ページ独立の静的配信）
- **ビルド成果物**
  - 各ルートの `.html` と、島用の外部 JS チャンク、静的アセット
- **デプロイ**
  - 生成した静的成果物を Cloudflare（Pages / Workers Assets）へデプロイできる

#### 非機能要件
- **インラインスクリプトを一切生成しない**こと（本プロジェクトの根幹）
- `unsafe-inline` を含まない CSP（最低でも `script-src 'self'`）で動作すること
- 純粋な静的サイト（MPA）として動作し、配信時にサーバーランタイムを必要としない
- 島が1つもないページは JS ゼロで配信できること
- ビルドが個人ブログ規模で十分高速であること
- 無料枠内で運用可能なデプロイ構成であること

#### 技術的制約
- `vite` + `react` + `react-dom` を前提技術とする。`@vitejs/plugin-rsc` には依存しない（RSC / Flight を使わないため、実験的依存による破壊的変更リスクを回避できる）
- **最重要の検証項目**: Vite のビルド出力（モジュールプリローダ・ランタイムローダ等）が**インラインスクリプトを残さない**ことを担保できるか。Astro が解決できなかった「ごくわずかなインライン」を Cirro が本当にゼロにできるかが成否を分けるため、最優先でプロトタイプ検証する
- props を `data-*` 属性で渡す際の適切なエスケープ（XSS 対策）に注意する
- スタイルについても、可能な範囲で `style-src` の厳格化（インラインスタイルの排除）を視野に入れる（要検証）

#### リソース制約
- 開発予算：ポケットマネー範囲内
- サーバーコストは最小限に抑える（無料枠内を目指す）

### 2.2 Phase 2 以降（将来）

純粋な静的 MPA で完結する範囲を Phase 1 とし、以下は将来オプションとして追加する。

- **メタデータ / SEO**
  - `<head>` 管理（title・meta）
  - OGP、サイトマップ、RSS の生成
- **SPA 的なクライアント遷移**（オプション）
  - ページ遷移をクライアント側で行うモード。Phase 1 の MPA をデフォルトに保ったまま、選択肢として追加する
- **外部コンテンツ連携**
  - Headless CMS や Cloudflare D1 などに登録したコンテンツを取得してビルド
- **動的・ステートフルな連携**
  - サーバー関数や Cloudflare Workers / Durable Objects 連携（コメント、カウンター等）
  - `@cloudflare/vite-plugin` 経由のエッジ連携、静的ページと動的ページのハイブリッド構成
- いずれの拡張でも、Phase 1 で確立した「インラインスクリプトゼロ / 厳格 CSP」の原則を崩さないことを前提とする

### 2.3 スコープ外の項目

**対応しない項目**
- ファイルベースルーティング
- React 以外のフレームワーク対応（React 特化を貫き、薄さと学習コストの低さを優先する）
- **MDX**
  - MDX はコンテンツ内に任意の JSX（＝実行可能コード）を埋め込めるため、セキュリティ面の不安がある
  - Cirro は基本的に静的サイトであり、コンテンツを実行可能にしないことでセキュリティを万全にできるとの考えから、利便性よりも安全性を優先して MDX を採用しない
  - リッチな表現はプレーン Markdown と島（`<Island>`）の組み合わせで実現する

## 3. 前提条件と制約条件

### 3.1 前提条件
- 開発者は個人（1人）で、Vtuber 活動と並行して開発を進める
- 開発過程を週一の動画コンテンツとして公開予定
- スモールスタートで、段階的に機能を追加していく方針
- 開発者は Astro を使用した経験があり、その CSP（`unsafe-inline` 必須）の制約への不満が本プロジェクトの出発点である

### 3.2 制約条件

#### 技術的制約
- 前述（2.1 技術的制約）のとおり、Vite ビルド出力からインラインスクリプトを完全排除できることの検証を最優先とする

#### リソース制約
- 開発予算：ポケットマネー範囲内

#### 運用制約
- サーバーコストは最小限に抑える（無料枠内を目指す）
- 開発ペースは無理のない範囲で調整（週単位の小さなマイルストーン）
- メンタルヘルスを最優先し、できない週があっても問題なしとする
