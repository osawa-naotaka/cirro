# プロジェクト憲章 - Cirro

## 1. プロジェクト基本情報

### 1.1 プロジェクト名
**Cirro** - React Server Components を使った軽量な static site generator (SSG)

### 1.2 プロジェクト実行の背景・課題

#### 現状の課題
- Next.js などの SSG は規模が大きく、簡単なサイトを構築するには複雑すぎる
- Vite 環境で React Server Components (RSC) を使った SSG はあまり見当たらない
- 一部だけ Cloudflare Workers / Durable Objects と連携をしたい（将来）

### 1.3 プロジェクトの目的

#### プロジェクトの目的
- 軽量な SSG を blog やポートフォリオサイトを運営したい人に提供する
- 静的サイトを Cloudflare にデプロイすることを第一目標とする
- クライアント側でもリアクティブなスクリプトを使えるようにする

#### 「軽量」の定義
本プロジェクトにおける「軽量」とは、**クライアントへ配信される JavaScript のバイト数が最小**という意味ではない（RSC は React ランタイムと Flight ペイロードをクライアントに必要とするため、その観点では純粋な静的サイトより重い）。

Cirro が目指す「軽量」は次の **開発体験 (DX) の軽さ** を指す:
- Next.js に比べて設定・概念がシンプルで、認知負荷が低い
- ビルドが速い
- フレームワークが提供する規約が少なく、把握しやすい

#### 概要
- Vite ベースで実装する。Cirro 自体は単一の Vite プラグインではなく、`@vitejs/plugin-rsc` を内包し、独自の SSG プラグイン・CLI・規約を束ねた薄いフレームワークとして提供する
- TypeScript を利用して型を活用する（特にルーティング設定の型付けを売りとする）
- React Server Components を利用して SSG を実現する
- `use client` ディレクティブを利用して、クライアント側で動作するスクリプトも同様に記述できるようにする

#### 技術的な土台（参考実装）
`@vitejs/plugin-rsc` の SSG サンプル（`examples/ssg`）の仕組みを土台とする:
- **3 つの Vite 環境** で動作する
  - `rsc`: Server Components の描画・リクエスト処理
  - `ssr`: Flight ストリームを HTML に変換（`react-dom/server.edge`）
  - `client`: ブラウザ側のハイドレーションとナビゲーション
- ビルド時に SSG プラグインが以下を実行する
  1. RSC エントリの `getStaticPaths()` を呼び、生成すべき全ルートを取得する
  2. 各パスについて `handleSsg()` に合成した `Request` を渡し、HTML と RSC ストリームを得る
  3. `/page` → `page.html` を、クライアント遷移用に `_.rsc`（シリアライズされた Flight データ）を書き出す
- **Cirro 本体は RSC ランタイムを自作しない。** `@vitejs/plugin-rsc` に依存し、その上に Config Base Routing・コンテンツ層・デプロイ設定を提供することに専念する。薄い皮として、便利機能を過剰に抱え込まず最小限の規約に留める
- なお参考実装は MDX を採用しているが、Cirro は後述（2.3）の理由により MDX を採用せず、プレーンな Markdown のみを扱う

## 2. プロジェクトスコープ

スコープは段階的に拡張する。Phase 1 を確実に完成させることを最優先とする。

### 2.1 Phase 1（初期バージョン・第一目標）— 完全静的 SSG

#### 機能要件
- **Config Base Routing**
  - `.ts` ファイルに JavaScript オブジェクトとしてルーティング情報を直接書き込む
  - ルート定義オブジェクトに型を付け、型安全なルーティングを提供する
  - 動的ルート（例: `/posts/[slug]`）は、ルート定義からパラメータ一覧を返す仕組み（`getStaticPaths` 相当）で表現する
- **コンテンツ層**
  - プレーンな Markdown を記事コンテンツとして扱えるようにする（MDX は採用しない。理由は 2.3 を参照）
- **開発体験**
  - dev サーバーと HMR
- **クライアント動作**
  - `use client` による Client Component とハイドレーション
- **ビルド成果物**
  - 各ルートの `.html`、クライアント遷移用の `.rsc`、および静的アセット
- **デプロイ**
  - 生成した静的成果物を Cloudflare（Pages / Workers Assets）へデプロイできる

#### 非機能要件
- 純粋な静的サイトとして動作し、配信時にサーバーランタイムを必要としない
- ビルドが個人ブログ規模で十分高速であること
- 無料枠内で運用可能なデプロイ構成であること
- セキュリティに配慮すること
  - CSPを設定

#### 技術的制約
- `@vitejs/plugin-rsc` は実験的で変化が速く、Next.js 外の RSC は最先端領域である。破壊的変更のリスクに備え、**依存バージョンを固定**して運用する
- React Server Components / Vite / Cloudflare を前提技術とする

#### リソース制約
- 開発予算：ポケットマネー範囲内
- サーバーコストは最小限に抑える（無料枠内を目指す）

### 2.2 Phase 2 以降（将来）— 一部サーバー実行 / 動的連携

純粋な SSG は静的ファイルのみで完結し Workers / Durable Objects を必要としない。動的・ステートフルな機能（コメント、カウンター、リアルタイム等）を扱うには、SSG とは別レイヤーの「一部サーバー実行」が必要になる。これは初期スコープ外とし、将来オプションとして追加する。

- **メタデータ / SEO**
  - `<head>` 管理（title・meta）
  - OGP、サイトマップ、RSS の生成（blog 用途の基盤）
- Headless CMSやClodflare D1などに登録したMarkdownファイルを取得してビルド
- `use server` / サーバー関数の対応
- `@cloudflare/vite-plugin` 経由での Cloudflare Workers / Durable Objects 連携
- 静的ページと動的ページのハイブリッド構成

### 2.3 スコープ外の項目

**対応しない項目**
- ファイルベースルーティング
- **MDX**
  - MDX はコンテンツ内に任意の JSX（＝実行可能コード）を埋め込めるため、セキュリティ面の不安がある
  - Cirro は基本的に静的サイトであり、コンテンツを実行可能にしないことでセキュリティを万全にできるとの考えから、利便性よりも安全性を優先して MDX を採用しない
  - リッチな表現はプレーン Markdown と `use client` の Client Component の組み合わせで実現する

## 3. 前提条件と制約条件

### 3.1 前提条件
- 開発者は個人（1人）で、Vtuber 活動と並行して開発を進める
- 開発過程を週一の動画コンテンツとして公開予定
- スモールスタートで、段階的に機能を追加していく方針

### 3.2 制約条件

#### 技術的制約
- 前述（2.1 技術的制約）のとおり、`@vitejs/plugin-rsc` のバージョン固定方針を採る
- 実験的な依存先の破壊的変更が開発ペースを乱すリスクを常に念頭に置く

#### リソース制約
- 開発予算：ポケットマネー範囲内

#### 運用制約
- サーバーコストは最小限に抑える（無料枠内を目指す）
- 開発ペースは無理のない範囲で調整（週単位の小さなマイルストーン）
- メンタルヘルスを最優先し、できない週があっても問題なしとする
