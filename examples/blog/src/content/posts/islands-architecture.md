## なぜ「島」なのか

多くのページは、その大半が静的なコンテンツです。にもかかわらず、ページ全体を
ハイドレートすると、本文のコードもランタイムもすべてクライアントへ送られてしまいます。

島（islands）アーキテクチャは、この発想を逆転させます。

- **本文は静的 HTML** として配信する（JS を送らない）
- **インタラクティブな箇所だけ**を「島」として切り出し、その部分だけハイドレートする

## Island の宣言

Cirro では、インタラクティブなコンポーネントを `<Island>` で囲むだけです。

```tsx
import { Island } from "../islands/Island";

export function Page() {
    return (
        <body>
            <p>この本文は静的 HTML です。</p>
            <Island name="counter" props={{ initial: 3 }} />
        </body>
    );
}
```

ビルド時、島は次の 3 つに展開されます。

| 要素 | 役割 |
| --- | --- |
| プレースホルダ要素 | ハイドレート先の DOM |
| `data-*` 属性 | props を JSON で受け渡す（インラインスクリプト不要） |
| `<script src>` | 島マウンタを外部ファイルとして読み込む |

ポイントは、**props をインラインスクリプトではなく `data-*` 属性で渡す**ことです。
これにより、初期状態を渡すためのインラインスクリプトが不要になり、CSP 厳格性を保てます。

## 島ゼロのページは JS ゼロへ

島を 1 つも置かなければ、そのページではハイドレーションが発生しません。たとえばこのブログの
**About ページやタグ一覧ページ**は、島を含まない静的なページです。

こうした島ゼロのページには本来マウンタも不要であり、**JS ゼロで配信できる**ことが Cirro の
目指すところです（島の有無に応じて配信を最適化する仕組みは今後の課題です）。

静的でいられる部分は静的なまま。動かしたい部分だけにコストを払う。それが島の考え方です。
